# myrmdb 三人分工

本文档用于规划三名成员在项目实现、汇报和报告中的职责。原则是总体任务量接近，同时满足 A 只负责项目实现、不负责汇报与报告。

## 分工原则

- A：承担更重的核心代码实现任务，不参与汇报和报告。
- B：承担中等偏重的代码实现任务，同时负责报告主线和技术文字。
- C：承担中等偏重的代码实现任务，同时负责汇报材料、演示和最终整合。
- 三人都需要参与代码评审和阶段验收，但 A 的非代码工作仅限于说明自己实现部分的技术细节。

## 工作量估算

| 成员 | 项目实现 | 汇报 | 报告 | 综合工作量 |
| --- | --- | --- | --- | --- |
| A | 高 | 无 | 无 | 约 1/3 |
| B | 中 | 辅助 | 主责 | 约 1/3 |
| C | 中 | 主责 | 辅助 | 约 1/3 |

## A：存储、执行核心与高风险底座

### 项目实现职责

- 题目一：存储管理主责。
  - `DiskManager` 文件创建、打开、关闭、删除、页读写。
  - `LRUReplacer` 替换策略。
  - `BufferPoolManager` 页获取、新建、换出、删除、刷盘。
  - `RmFileHandle` 记录插入、读取、删除、更新。
  - `RmScan` 表扫描。
- 题目二：查询执行核心算子主责。
  - `SeqScanExecutor`
  - `ProjectionExecutor`
  - `NestedLoopJoinExecutor`
  - `UpdateExecutor`
  - `DeleteExecutor`
- 题目八：块嵌套循环连接主责。
  - join buffer 设计。
  - 大数据量连接。
  - 非等值连接。

### 交付物

- 存储管理单元测试通过。
- 基础查询执行测试通过。
- BNLJ 大数据连接测试通过。
- 为 B/C 提供实现说明要点，但不撰写报告正文、不制作汇报材料。

### 不负责事项

- 不负责汇报 PPT。
- 不负责报告撰写。
- 不负责答辩讲述。

## B：语义分析、类型系统、报告主线

### 项目实现职责

- 题目二：语义分析和元数据管理主责。
  - `SmManager::open_db`
  - `SmManager::close_db`
  - `SmManager::drop_table`
  - `Analyze` 中表存在、列存在、类型检查。
  - UPDATE 语义分析。
  - INSERT、DELETE、SELECT 的错误处理补齐。
- 题目三：BIGINT 类型主责。
  - parser 类型和字面量扩展。
  - `Value`、`ColType`、输出、比较和存储扩展。
  - 溢出检测。
- 题目四：DATETIME 类型主责。
  - parser 类型扩展。
  - 时间编码和格式化。
  - 日期合法性校验。
  - DATETIME 条件比较。
- 题目六：聚合函数协作。
  - 负责 parser、AST、Analyze、Plan 层表达。

### 报告职责

- 报告主责。
- 负责报告结构设计：
  - 项目背景与总体架构。
  - 模块设计。
  - 关键数据结构。
  - 测试方法。
  - 问题与解决方案。
- 负责整合 A/C 提供的技术要点。
- 负责统一报告术语、图表说明、代码路径引用。

### 汇报职责

- 辅助 C 梳理技术讲述逻辑。
- 提供类型系统、语义分析、元数据管理相关讲稿要点。

### 交付物

- 题目二中元数据和语义检查稳定通过。
- BIGINT、DATETIME 功能测试通过。
- 聚合函数前端表达可用。
- 完整报告初稿和终稿。

## C：索引、事务恢复、汇报主线

### 项目实现职责

- 题目五：唯一索引主责。
  - B+ 树补全。
  - `create index`、`drop index`。
  - `show index from table_name`。
  - 唯一性约束。
  - 索引查询和范围查询。
  - 插入、删除、更新时维护索引。
- 题目六：聚合函数协作。
  - 负责聚合执行器或聚合执行逻辑。
  - 负责 COUNT、SUM、MAX、MIN 输出格式。
- 题目七：ORDER BY 和 LIMIT 主责。
  - 多排序键。
  - ASC/DESC。
  - LIMIT。
  - `SortExecutor` 改造。
- 题目九：事务控制语句主责。
  - begin、commit、abort。
  - 显式事务和单语句事务区分。
  - 回滚 DML。
- 题目十：并发控制主责。
  - 两阶段封锁。
  - no-wait 死锁预防。
  - 表级锁和记录级锁。
  - 并发事务异常测试。
- 题目十一：故障恢复主责。
  - WAL。
  - redo/undo 日志。
  - 崩溃重启恢复。

### 汇报职责

- 汇报主责。
- 负责 PPT 和演示流程：
  - 项目整体架构。
  - 完成题目列表。
  - 核心模块演进。
  - 难点与亮点。
  - 测试结果展示。
- 负责准备演示 SQL 和运行脚本。
- 负责答辩问题预案。

### 报告职责

- 辅助 B 完成索引、事务、并发、恢复章节。
- 提供测试截图、输出样例和性能对比材料。

### 交付物

- 唯一索引、聚合执行、排序限制、事务、并发、恢复功能通过测试。
- 汇报 PPT。
- 演示脚本和答辩提纲。

## 协作接口

### A 与 B

- A 完成记录扫描和执行器基础接口后，B 接入语义分析和计划生成。
- B 新增 BIGINT、DATETIME 后，需要同步字段长度、比较函数和输出格式给 A 的执行器逻辑。

### A 与 C

- A 的扫描、连接、更新、删除算子是 C 实现索引维护、BNLJ、事务回滚的基础。
- C 修改索引和事务逻辑时，需要确认不会破坏 A 的记录文件和缓冲池不变量。

### B 与 C

- B 负责 parser/analyze 中新增语法和语义表达，C 负责对应执行逻辑。
- 聚合、ORDER BY、LIMIT、索引展示等功能需要 B/C 明确 AST、Plan、Executor 的字段边界。

## 阶段里程碑

### 阶段一：底座可用

- A 完成题目一。
- B 完成数据库打开关闭、删表、基础语义检查。
- C 阅读索引和事务恢复代码，准备后续设计。

验收标准：

- `unit_test` 通过。
- 服务端可启动。
- 基础建表、插入、查询链路可进入执行层。

### 阶段二：基础 SQL 可用

- A 完成扫描、投影、连接、更新、删除执行器。
- B 完成题目二语义分析和元数据管理。
- C 协助补齐输出格式和 SQL 回归测试。

验收标准：

- 题目二五个测试点通过。

### 阶段三：类型、索引、聚合、排序

- B 完成 BIGINT、DATETIME。
- C 完成唯一索引、聚合函数、ORDER BY/LIMIT。
- A 完成 BNLJ。

验收标准：

- 题目三到题目八通过。
- 索引查询有明显性能收益。

### 阶段四：事务、并发、恢复

- C 主导事务控制、并发控制、故障恢复。
- A 支持底层记录、缓冲池、执行器层面的回滚和日志接入。
- B 支持报告整理和事务恢复章节说明。

验收标准：

- 题目九到题目十一通过。
- 崩溃恢复后基表和索引一致。

### 阶段五：最终交付

- A：完成代码自查和核心实现说明。
- B：完成最终报告。
- C：完成最终汇报和演示。

验收标准：

- 全量构建通过。
- 已完成题目测试通过。
- 报告、PPT、演示 SQL 与代码版本一致。

- [ ] 当前已完成题目是否有回归测试。
- [ ] 是否有输出格式变化。
- [ ] 是否有未同步的 parser 生成文件。
- [ ] 是否有影响其他成员接口的结构体或函数签名变化。
- [ ] 是否有需要写入报告或汇报的设计变更。
