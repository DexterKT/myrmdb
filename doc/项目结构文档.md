# **RMDB**项目结构详解

## **代码框架**

系统包括查询解析、查询优化、查询执⾏、索引、事务、⽇志、存储等⼏个模块，当系统通过tcp连接接收到客户端发送的⼀条SQL语句时，⾸先通过parser进⾏解析，得到⼀个抽象语法树（AST），接着通过analyze解析器进⾏语义检查⽣成query tree（对应代码中的Query类），然后进⼊optimizer模块的plan_query()函数进⾏查询优化⽣成查询执⾏计划（对应代码中的Plan类），接着进⼊到portal模块进⾏查询执⾏，在portal模块中，⾸先通过start()函数进⾏相关资源的初始化，并把查询执⾏计划转换成算⼦树，根据算⼦树调⽤不同的算⼦以及system（元数据管理器）进⾏查询执⾏。在执⾏过程中，需要和事务管理器交互来进⾏事务资源的管理，和锁管理器交互对并发事务进⾏正确性控制，和存储模块进⾏交互进⾏数据的读写，以及和⽇志管理器交互添加⽇志。

### 1. 全局核心流水线 (主线数据流)
SQL: string -> parser -> [ast] -> analyze -> [query tree] -> optimizer.plan_query -> optimizer.plan -> portal.run -> executor -> nextTuple

---

### 2. 模块内部拓扑与详细流向

#### 模块 A: 优化器 (optimizer)
* **入口**: `plan_query`
* **分流路由**: `plan_query` 静态分流至 5 个并列的 SQL 类型节点：
  - SELECT
  - INSERT
  - UPDATE
  - DELETE
  - DDL
* **SELECT 特殊子流程** (自上而下串行): 
  SELECT -> [基表生成plan] -> [根据连接条件生成join] -> [将剩余基表连接] -> [生成投影] -> plan
* **汇聚出口**: INSERT / UPDATE / DELETE / DDL 以及 SELECT 子流程的末端，最终全部汇聚并输出至 `plan` 节点。

#### 模块 B: 入口分发器 (portal)
* **入口**: 接收来自优化器的 `plan`
* **内部节点与交互**:
  - `start`: 向上触发两个并列动作：[处理不同的plan类型] 和 [根据plan生成执行算子]
  - `run`: 核心控制路由，向外引出三条分支：
    1. 向上流向事务模块：`TransactionManager`（带有控制标签: Begin/Commit/Abort）
    2. 向右流向执行器：`executor`
    3. 向下流向系统模块：`system`（带有控制标签: create/drop table/index）
  - `drop`: 独立的资源清理节点

#### 模块 C: 执行器与支撑系统 (executor & backend)
* **核心驱动**: `executor`
  - 向右通过迭代器模型调用 `nextTuple`
  - 向下将元数据（标签: meta）传递给 `system`
* **系统与索引管理 (system & index)**:
  - `system` 向下连接 `index` -> `index` 向下连接基于 `B+ tree` 的具体实现
  - `system` 横向向右连接存储层的 `DiskManager`

#### 模块 D: 事务与锁管理器 (transaction)
* **内部结构**: `TransactionManager` -> 纵向向下控制 `LockManager`
* **对外约束**: `LockManager` 向下对 `nextTuple` 施加锁（标签: Lock）

#### 模块 E: 存储引擎 (storage)
* **数据流向**: `nextTuple` 底层通过元组数据（标签: tuple）与存储层交互。
* **存储依赖链** (自上而下纵向依赖): 
  `RmFileHandle` (记录文件句柄) -> `BufferPool` (缓冲池) -> `DiskManager` (磁盘管理器)

#### 模块 F: 日志管理器 (LogManager)
* **日志写入流**: 
  `nextTuple` 运行中产生日志（标签: Log） -> 流向 `AddLogToBuffer` -> 纵向向下流向 `FlushLog`
* **持久化流向**: `FlushLog` 最终将日志数据刷入最底层的 `DiskManager`

## **存储管理**

### **相关知识点**

数据库存储基本原理：数据库存储模型、存储介质和磁盘管理

⽂件存储组织：堆⽂件、顺序⽂件、B+树⽂件、Hash⽂件

元数据存储组织：数据字典；数据字典的组织与存储

记录存储组织：定⻓记录组织

缓冲区管理：缓冲区组织⽅式，缓冲区管理策略

### **项目结构**

存储管理模块相关的⼦⽬录为src/record、src/replacer、src/storage和src/system，分别对应记录存储组织、缓冲区管理、⽂件存储管理和元数据存储组织。存储管理模块是底层模块，主要为上层模块提供接⼝。在RMDB中，每⼀个数据库对应⼀个同名⽂件夹，该数据库中的所有数据都以⽂件的形式存储在该⽂件夹中。每⼀个数据库包含⼀个 <database>.meta ⽂件和 <database>.log ⽂件， <database>.meta ⽂件⽤来存储数据库的元数据，包括表的名称、表的元数据等信息； <database>.log ⽂件⽤来存储⽇志信息。
RMDB使⽤关系模型来进⾏存储，数据库中的表 <table> 包含了若⼲条记录 <record> ，这些记录都顺序存储在该表的同名⽂件中，并且以数据⻚(Page)的形式来对所有记录分⻚，进⽽通过对数据⻚的管理来统⼀管理表中的数据。


以下对各个⼦模块进⾏详细介绍：

#### ⽂件存储组织模块：src/storage

RMDB以数据⻚(Page)为单位对内存中的数据来进⾏统⼀管理。系统提供了Page类(Page class)，通过BufferPoolManager来管理数据⻚在磁盘和内存中的I/O，通过DiskManager来进⾏磁盘中⽂件的读写。

**Page**

Page类是数据管理的基本单位，每⼀个Page都对应唯⼀的⻚号(PageId)，每⼀个数据⻚都对应⼀个磁盘⽂件，对于同⼀个⽂件中的数据⻚，⽤page_no来对数据⻚进⾏索引。

```cpp
struct PageId {
 int fd; // Page所在的磁盘⽂件开启后的⽂件描述符, 来定位打开的⽂件在内存中的位置
 page_id_t page_no = INVALID_PAGE_ID; // 该数据⻚位于所在⽂件的第⼏个数据⻚
};
```

每⼀个Page对象中存储了该数据⻚的相关信息以及该数据⻚存储的数据，可以分为⻚头(header)和数据区(data)，⻚头存放有关⻚⾯内容的元数据；数据区是存放⻚⾯实际数据的区域，它可以⽤于存放表中的记录。由于⻚⾯既可以⽤于存储记录，也可以⽤于存储索引，所以⻚头和数据区这两部分的实际内容其实是由记录的组织结构或索引的组织结构决定的。为了区分这两者，对于记录⻚，定义为由记录头（record page header）和数据区组成，对于索引⻚，定义为由索引头（index page header）和数据区组成。⽬前本系统将⻚⾯⼤⼩（PAGE_SIZE）设计为固定的4KB⼤⼩，⻚⾯的数据结构如下：

```cpp
class Page {
 public:
 Page() { ResetMemory(); }
 private:
 void ResetMemory() { memset(data_, OFFSET_PAGE_START, PAGE_SIZE); } // 将data_的
PAGE_SIZE个字节填充为0
 PageId id_; /** page的唯⼀标识符 */
 char data_[PAGE_SIZE] = {}; // 该数据⻚存储的数据
 bool is_dirty_ = false; // 该数据⻚是否为脏⻚，如果是脏⻚，系统需要根据缓冲池管理策略
在合适的时间将⻚⾯数据写⼊磁盘
 int pin_count_ = 0; // ⽤于记录正在使⽤该⻚⾯的线程个数
 ReaderWriterLatch rwlatch_; // 该数据⻚的latch，⽤来保证数据⻚的并发操作正确性。
 /*
 ReaderWriterLatch分为读锁和写锁，每个线程在读写⻚⾯之前需要申请对应的锁，操作完成后释放对应的锁。
如果在⼀个线程中获取了⻚⾯的写锁，那么其他线程⽆法读取和写⼊该⻚⾯，直到其释放写锁；如果在⼀个线程中获取
了⻚⾯的读锁，那么其他线程⽆法写⼊该⻚⾯（可以读取），直到其释放读锁。
 */
};
```

在这些数据结构中，只有data是在内存和磁盘中进⾏交换，其内容就是之前提到的⻚头和数据区。

其他的数据结构只在内存中使⽤，并不会写⼊磁盘。

**DiskManager**

DiskManager负责磁盘⽂件的操作，为上层模块提供了操作⽂件、读写⻚⾯的功能。其数据结构如下：

```cpp
class DiskManager {
 private:
 std::unordered_map<std::string, int> path2fd_; // <Page⽂件磁盘路径,Page fd>哈希表
 std::unordered_map<int, std::string> fd2path_; // <Page fd,Page⽂件磁盘路径>哈希表
 int log_fd_ = -1; // ⽇志⽂件的句柄
 std::atomic<page_id_t> fd2pageno_[MAX_FD]{}; // 在⽂件fd中分配的page no个数
};
```

**BufferPoolManager**

BufferPoolManager对内存中的缓冲池进⾏管理，内存中的缓冲池由若⼲个帧(frame)组成，缓冲池中每⼀个帧可以存放⼀个数据⻚，没有存放数据⻚的帧叫做空闲帧。系统对缓冲池中所有数据⻚的修改都暂时保存在内存中，只有当缓冲池中的数据⻚被替换时，才会把这些修改的数据刷新到磁盘中。缓冲池数据结构如下：

```cpp
class BufferPoolManager {
 private:
 size_t pool_size_; // 缓冲池中帧的数⽬，也就是最多能够同时容纳的数据⻚的个数
 Page *pages_; // 连续存放的⻚⾯数组，在构造函数中进⾏初始空间申请
 std::unordered_map<PageId, frame_id_t, PageIdHash> page_table_; // ⻚表，记录数据⻚和帧
的映射关系，如果数据⻚存放在某⼀帧中，则在⻚表中存放该数据⻚ID和对应帧号的映射关系，⽤于标记数据⻚在缓冲
池中的存放位置
 std::list<frame_id_t> free_list_; // 缓冲池中未使⽤的空闲帧的帧号列表
 LRUReplacer *replacer_; // ⽤于提供缓冲池的⻚⾯替换算法。当空闲帧列表为空时，表示所有的帧都被⻚
⾯占⽤了，这时如果要把⼀个新的数据⻚放⼊缓冲池中，就必须淘汰⼀个旧⻚⾯，使⽤替换算法器可以选择⼀个能被淘
汰的⻚⾯。
 std::mutex latch_; // ⽤于缓冲池的并发控制
};
```

#### **缓冲区管理：**src/replacer

缓冲区管理模块主要提供了缓冲池⻚⾯的管理策略，⽬前本系统使⽤LRU(Least Recently Used)管理策略，即最近最少使⽤⻚⾯置换算法，当缓冲池需要淘汰⼀个⻚⾯时，选择最近最久未使⽤的⻚⾯进⾏淘汰，提供以下数据结构：

```cpp
class LRUReplacer {
 private:
 std::mutex latch_; // 互斥锁，⽤于并发控制
 std::list<frame_id_t> LRUlist_; // 维护不被固定的帧列表，不被固定的帧存放的⻚⾯是可以被淘汰
的，该列表按照帧被取消固定的时间戳顺序存放
 std::unordered_map<frame_id_t, std::list<frame_id_t>::iterator> LRUhash_; // 维护帧
号和LRUlist_中对象的映射关系
 size_t max_size_; // 最⼤容量（与缓冲池的容量相同）
};
```

#### **记录存储组织模块：**src/record

记录存储组织模块为上层模块提供了对表中记录的操作接⼝，包括记录⽂件的创建和删除、记录的插⼊修改和删除、记录⽂件中数据⻚的读写等。⾸先介绍记录⽂件相关的数据结构，每⼀个数据库中的表可以分为表的元数据和表的记录数据，表的元数据统⼀存储在 <database.meta> ⽂件中，表的记录数据单独存储在同名⽂件中，我们称该⽂件为记录⽂件，在RMDB中，我们⽤RmFileHandle类来对⼀个记录⽂件进⾏管理，RmFileHandle类中记录了⽂件描述符和⽂件头信息，数据结构如下：

```cpp
class RmFileHandle {
 private:
 int fd_; // ⽂件描述符，操作系统打开磁盘⽂件后的⼀个编号，⽤于唯⼀地标识打开的⽂件
 RmFileHdr file_hdr_; // ⽂件头信息
};
```

⽂件头信息⽤RmFileHdr结构体来进⾏维护。前⽂中提到，记录⽂件中顺序存放该表的每⼀条记录，按照数据⻚的形式进⾏划分，在⽬前系统中，同⼀个表中的所有记录⼤⼩是相同的，因此，在⼀个表创建之初，该表的record_size就固定下来了；对于数据⻚的管理，本系统⽬前使⽤Bitmap的形式进⾏管理，因此，在RmFileHdr中还记录了bitmap的⼤⼩，对于每⼀个数据⻚，都使⽤⼀个Bitmap来对该数据⻚的记录进⾏管理，因此Bitmap的⼤⼩在表创建之初就可以计算出来。RmFileHdr的数据结构如下：

```cpp
struct RmFileHdr {
 int record_size; // 元组⼤⼩（⻓度不固定，由上层进⾏初始化）
 int num_pages; // ⽂件中当前分配的page个数（初始化为1）
 int num_records_per_page; // 每个page最多能存储的元组个数
 int first_free_page_no; // ⽂件中当前第⼀个可⽤的page no（初始化为-1）
 int bitmap_size; // bitmap⼤⼩
};
```

file_hdr中的num_pages记录此⽂件分配的page个数，page_no范围为[0,file_hdr.num_pages)，page_no从0开始增加，其中第0⻚存file_hdr，从第1⻚开始存放真正的记录数据。

对于每⼀个数据⻚，本系统使⽤RmPageHandle类进⾏封装管理，每⼀个数据⻚的开始部分并不直接存放记录数

据，⽽是按照以下顺序进⾏存放：

| page_lsn_ | page_hdr | bitmap | slots |

- page_lsn_⽤于故障恢复模块，将在故障恢复模块中进⾏详细介绍。

- page_hdr记录了两个信息，⼀个是num_records，记录当前数据⻚中已经分配的record个数，同时记录了

- next_free_page_no，记录了如果当前数据⻚写满之后，下⼀个还有空闲空间的数据⻚的page_no。

- bitmap记录了当前数据⻚记录的分配情况。

- slots则是真正的记录数据存放空间。

RmPageHandle的数据结构如下：

```cpp
struct RmPageHandle {
 const RmFileHdr *file_hdr; // ⽤到了file_hdr的bitmap_size, record_size
 Page *page; // 指向单个page
 RmPageHdr *page_hdr; // page->data的第⼀部分，指针指向⾸地址，⻓度为
sizeof(RmPageHdr)
 char *bitmap; // page->data的第⼆部分，指针指向⾸地址，⻓度为file_hdr-
>bitmap_size
 char *slots; // page->data的第三部分，指针指向⾸地址，每个slot的⻓度为
file_hdr->record_size
};
```

对于具体的每⼀个记录，使⽤RmRecord来进⾏管理，数据结构如下：

```cpp
struct RmRecord {
 char *data; // data初始化分配size个字节的空间
 int size; // size = RmFileHdr的record_size
};
```

同时，使⽤Rid来对每⼀个记录进⾏唯⼀的标识，数据结构如下：

```cpp
struct Rid {
 int page_no; // 该记录所在的数据⻚的page_no
 int slot_no; // 该记录所在数据⻚中的具体slot_no
};
```

#### **元数据存储组织：src/system**

system模块主要负责元数据管理和DDL语句操作执⾏，在存储模块主要涉及到了元数据管理，因此只对元数据存储相关的数据结构进⾏介绍，其余功能在查询模块进⾏介绍。

元数据存储相关的数据结构主要在sm_meta.h⽂件中，⾸先，对于⼀个数据库，使⽤DbMeta类来进⾏数据库元数据管理，数据结构如下：

```cpp
class DbMeta {
 std::string name_;
 std::map<std::string, TabMeta> tabs_;
};
```

主要包含数据库的名称，以及数据库终所有表的元数据，这些数据每执⾏⼀次DDL语句都要被写⼊到 <database.meta> ⽂件中，并且在open_db()时读取到内存中。

表的元数据主要通过TabMeta数据结构来进⾏管理：

```cpp
struct TabMeta {
 std::string name; // 表名称
 std::vector<ColMeta> cols; // 表包含的字段
 std::vector<IndexMeta> indexes; // 表上建⽴的索引
};
```

对于每⼀个字段，通过ColMeta数据结构来进⾏管理：

```cpp
struct ColMeta {
 std::string tab_name; // 表名称
 std::string name; // 字段名称
 ColType type; // 字段类型
 int len; // 字段⻓度
 int offset; // 字段所在记录中的偏移量，⽤于查询字段的具体存储位置
};
```

对于索引结构，通过IndexMeta数据结构来进⾏管理：

```cpp
struct IndexMeta {
 std::string tab_name; // 索引所属表名称
 int col_tot_len; // 索引字段⻓度总和
 int col_num; // 索引字段数量
 std::vector<ColMeta> cols; // 索引包含的字段
};
```

### 索引(Index)

#### **相关知识点**

顺序索引：稠密索引及其查找算法，稀疏索引及其查找算法，多级索引及其查找算法，辅助索引

B+树索引：B+树索引结构，索引查找，索引维护

哈希索引：基本Hash索引，可扩展Hash索引，线性Hash索引

Bitmap索引：Bitmap索引结构，Bitmap索引查找，编码Bitmap索引

#### **项⽬结构**

索引模块相关的⼦⽬录为src/index，主要实现了B+树，⽤来⽀持索引功能。

在本系统中，B+树以⽂件的⽅式存储在磁盘中，B+树的每个节点存储在⼀个磁盘⽂件的⻚⾯中，每个结点存储若⼲个键值对，在本系统中，如果想要把⼀个元组插⼊B+树，那么该元组会被解析成⼀个键值对 <key, value> ，其中key存储该元组中索引所在列的数据，value代表该元组的rid。

我们使⽤ IxFileHdr 来管理B+树⽂件的元数据，其中num_pages_⽤来记录⽂件中的⻚⾯数量，root_page_⽤来存储B+树根节点对应的⻚号，first_free_page_no_代表⽂件中第⼀个空闲的磁盘⻚⾯的⻚⾯号，col_num_记录索引包含的字段数量，col_types_和col_lens_分别记录字段的数据类型和数据⻓度，col_tot_len_存储索引包含的字段的总⻓度，btree_order_存储⼀个B+树节点所能存储的最多的键值对数量。在每个节点对应的⻚⾯中，⻚⾯⾸部存储⻚⾯的元信息，然后存储所有的key，⻚⾯的末尾部分⽤来存储所有的value，因此，需要⽤keys_size⽤来记录当前节点中所有key数据的⼤⼩，以便确定value部分的⾸地址。first_leaf和last_leaf分别存储⾸叶节点对应的⻚号和尾叶节点对应的⻚号。

```cpp
class IxFileHdr {
 page_id_t first_free_page_no_; // ⽂件中第⼀个空闲的磁盘⻚⾯的⻚⾯号
 int num_pages_; // 磁盘⽂件中⻚⾯的数量
 page_id_t root_page_; // B+树根节点对应的⻚⾯号
 int col_num_; // 索引包含的字段数量
 std::vector<ColType> col_types_; // 字段的类型
 std::vector<int> col_lens_; // 字段的⻓度
 int col_tot_len_; // 索引包含的字段的总⻓度
 int btree_order_; // # children per page 每个结点最多可插⼊的键值对数
量
 int keys_size_; // keys_size = (btree_order + 1) * col_tot_len
 // first_leaf初始化之后没有进⾏修改，只不过是在测试⽂件中遍历叶⼦结点的时候⽤了

始化，初始化为root page_no

```

系统使⽤ IxPageHdr 来管理每个磁盘块（在内存中叫⻚⾯）的元数据，其中parent记录⽗结点所在⻚⾯的⻚号，num_key代表当前节点中已经插⼊的键值对的数量，is_leaf⽤来判断当前节点是否为叶⼦节点，如果当前结点是叶⼦结点，那么prev_leaf存储当前结点的前⼀个叶⼦节点，next_leaf记录当前结点的下⼀个叶⼦结点。

```cpp

 page_id_t next_free_page_no; // unused
 page_id_t parent; // ⽗亲节点所在⻚⾯的叶号
 int num_key; // # current keys (always equals to #child - 1) 已插
⼊的keys数量，key_idx∈[0,num_key)
 bool is_leaf; // 是否为叶节点
 page_id_t prev_leaf; // previous leaf node's page_no, effective only
when is_leaf is true
 page_id_t next_leaf; // next leaf node's page_no, effective only when
is_leaf is true
};
```

系统在其他模块如存储模块、执⾏模块中，使⽤ Rid 来存储记录号，在索引模块中使⽤ Iid 来存储记录号，⼆者是等价的，⼀⼀对应的。

系统中， IxNodeHandle 和 IxIndexHandle 分别提供了对B+树节点的操作⽅法和对B+树的操作⽅法。 IxManager 类为上层模块提供了创建索引、删除索引、把索引⽂件读取到内存中、关闭索引⽂件等接⼝。 IxScan 类提供了遍历B+树叶⼦节点的功能。

### 故障恢复(Failure recovery)

#### **相关知识点**

基于REDO/UNDO⽇志的恢复算法：基于REDO/UNDO⽇志的恢复算法；算法的优缺点分析

恢复算法ARIES：LSN；⽇志结构；⽇志缓冲区管理；ARIES算法

#### **项⽬结构**

故障恢复模块相关代码⼦⽬录为src/recovery，主要包含⽇志管理器和故障恢复管理器。

⽇志记录使⽤LogRecord对象来进⾏存储，LogRecord分为⽇志头和⽇志数据项，⽇志头记录每⼀条⽇志都需要记录的固定信息，包括以下内容：

### 查询处理与执⾏(Query processing and execution)

#### **相关知识点**

查询解析：词法分析，语法分析，语法树

基本算⼦实现：扫描操作算法：全表扫描，索引扫描；连接操作算法：嵌套循环连接，Hash连接，索引连接

查询优化实现技术：查询优化的搜索空间，搜索优化计划的⽅法（穷举法、启发式⽅法、动态规划等）

查询执⾏框架：⽕⼭执⾏模型；物化执⾏模型；向量执⾏模型（SIMD）

#### **项⽬结构**

查询处理与执⾏模块相关的⼦⽬录为src/analyze、src/optimizer、src/execution、src/system。

本系统的查询引擎采⽤迭代器模型，每个数据库操作对应⼀个算⼦(executor)，每个算⼦提供⼀个核⼼接⼝Next()，执⾏⼀次算⼦便可通过Next接⼝获得⼀个元组。系统提供了insert操作对应的算⼦，参赛队伍需要⾃主实现update、delete对应的算⼦，以及扫描算⼦、连接算⼦等。

当SQL语句经过语法解析模块的处理，获得抽象语法树之后，进⼊分析器analyze，在分析器中需要进⾏语义分析，包括表是否存在、字段是否存在等，并把AST改写成Query，然后进⼊optimizer阶段，optimizer负责进⾏查询优化并⽣成查询执⾏计划，⽣成查询执⾏计划后，进⼊portal模块，portal模块分为start、run和drop三个阶段，start阶段负责相关资源初始化，并把查询执⾏计划转换成对应的算⼦树，run阶段通过各个算⼦的Next()接⼝进⾏sql语句的执⾏，drop阶段需要释放start阶段申请的资源。

对于DDL语句和事务语句，在进⼊到portal模块的run阶段后，会和SmManager以及TransactionManager进⾏交互，进⾏DDL语句及事务语句的执⾏。

### **语法解析**

语法解析模块相关代码位于src/parser⽂件夹中。

#### **语法树结构**

本系统的语法树中，不同类型的节点对应不同的结构体，所有节点都继承⽗类TreeNode，所有的语法树节点都定义在ast.h⽂件中，以下⾯的select语句为例，通过对该语句语法树的描述，来举例说明本系统语法树结构：

```cpp
SELECT R.id , S.date
FROM R JOIN S
ON R.id = S.id
WHERE S.value>100;
```

select语句对应的语法树结构：

以下是该 SQL 查询对应的 AST（Abstract Syntax Tree）逻辑拓扑及节点内部字段属性描述，采用强类型与指针引用关系进行建模。

##### 1. 核心根节点 (Root Node)
* **节点类型**: `SelectStmt`
* **成员变量与指针流向**:
  - `vector<shared_ptr>: target_list` -> 指向投影列链表 (第一个 `Column` 节点)
  - `vector<shared_ptr>: from_clause` -> 指向连接表达式节点 `JoinExpr`
  - `vector<shared_ptr>: where_clause` -> 指向条件表达式节点 `BinaryExpr`

---

##### 2. 语法树子节点与依赖流向

2.1 投影列表 (target_list 分支)
* **节点 1 (Column)**: 
  - 属性: `string: table_name("R")`, `string: field_name("id")`
  - 关系: 纵向虚线连接（代表链表或数组元素）至 节点 2
* **节点 2 (Column)**: 
  - 属性: `string: table_name("S")`, `string: field_name("date")`

2.2 数据源连接 (from_clause 分支)
* **核心节点**: `JoinExpr`
  - 属性: `JoinType: JOIN_INNER`, `bool: isNatural(false)`, `char*: alias(NULL)`
  - **指针分流**:
    1. `Node*: larg` (左参数) -> 指向 `Table` 节点 (属性: `catalog_name: (NULL)`, `table_name: ("R")`)
    2. `Node*: rarg` (右参数) -> 指向 `Table` 节点 (属性: `catalog_name: (NULL)`, `relation_name: ("S")`)
    3. `Node*: quals` (连接条件) -> 指向等值表达式节点 `A_Expr`

* **连接条件子树 (`A_Expr`)**:
  - 属性: `OpType: OP_EQ` (等号 `=` 操作)
  - **指针分流**:
    - `Node*: lexpr` (左表达式) -> 指向 `Column` (属性: `string: table_name("R")`, `char*: field_name("id")`)
    - `Node*: rexpr` (右表达式) -> 指向 `Column` (属性: `string: table_name("S")`, `char*: field_name("id")`)

 2.3 过滤条件 (where_clause 分支)
* **核心节点**: `BinaryExpr`
  - 属性: `OpType: OP_GT` (大于号 `>` 操作)
  - **指针分流**:
    1. `Node*: lexpr` (左表达式) -> 指向 `Column` (属性: `char*: relation_name("S")`, `char*: field_name("value")`)
    2. `Node*: rexpr` (右表达式) -> 指向 `IntValue` (属性: `long: value(10)`)

---

##### 3. 该 AST 逆向推导的原始 SQL 语句
```sql
SELECT R.id, S.date 
FROM R INNER JOIN S ON R.id = S.id 
WHERE S.value > 10;**
```

可以看到，该语法树的根节点指出整个语句为Select类型，其⼦节点包括了被选投影列节点列表，from从句和where从句列表。在from从句⼦树中，根节点为Join运算，⼦节点包括了需要连接的两个表R和表S以及连接条件表达式节点A_Expr，A_Expr节点和其叶⼦节点⼀起构成了连接运算条件R.id=S.id。⽽where从句节点同样也构成了⼀个表达式⼦树，通过叶⼦节点的数据和表达式的OpType:Greater构成了选择条件S.value>100。
